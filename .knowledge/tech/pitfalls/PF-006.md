# [PF-006] Pandas 时区 tz-naive 与 tz-aware 比较错误

> **类型**: pitfall
> **成熟度**: verified
> **创建时间**: 2026-07-24
> **最后引用**: 2026-07-24
> **标签**: python, pandas, timezone, tz-naive, tz-aware, datetime, multi-source, ercot, meteostat
> **适用阶段**: implement, debug

## 背景

多数据源联动分析中，不同数据源的 datetime 时区处理方式不同，混合使用 tz-naive（无时区）和 tz-aware（有时区）的 Timestamp 会导致 `TypeError: Cannot compare tz-naive and tz-aware datetime-like objects`。

## 陷阱 1: Meteostat 数据是 tz-naive

**现象**：Meteostat 小时数据 CSV 中的 `time` 列是纯 datetime 字符串，`pd.read_csv()` 后无时区信息。

**影响**：与其他 tz-aware 数据做比较（如 `df[df['time'] >= start]`）时报 TypeError。

**解决方案**：加载后立即统一为 tz-aware UTC：
```python
df["time"] = pd.to_datetime(df["time"])
if df["time"].dt.tz is None:
    df["time"] = df["time"].dt.tz_localize("UTC")
```

## 陷阱 2: ERCOT CSV 数据解析方式不统一

**现象**：GridStatus.io 下载的 ERCOT RTM 数据中 `interval_start_utc` 列带有 `+00:00` 后缀，`pd.to_datetime()` 自动解析为 tz-aware。而 EIA API 下载的 region_data/fuel_type_data 的 `time` 列不带时区后缀，解析为 tz-naive。

**影响**：同一个分析脚本中混用两类数据时，`.iloc[]` 返回的 numpy datetime64 对象也是 tz-naive，需要显式转换。

**解决方案**：在每个数据加载函数中统一处理，不依赖上游格式：
```python
df["time"] = pd.to_datetime(df["time"])
if df["time"].dt.tz is None:
    df["time"] = df["time"].dt.tz_localize("UTC")
```

## 陷阱 3: numpy datetime64 赋值到 dict 时丢失时区

**现象**：用 `df["time"].values[i]` 取出单值后存入 dict，该值是 numpy datetime64，没有 `.tz` 属性。

**影响**：后续用该 dict 值与 tz-aware DataFrame 做比较时报错。

**解决方案**：存入 dict 时显式转换并补时区：
```python
"time": pd.Timestamp(times[i]).tz_localize("UTC") if pd.Timestamp(times[i]).tz is None else pd.Timestamp(times[i])
```

## 陷阱 4: `pd.to_datetime(..., utc=True)` 会改变行为

**现象**：`utc=True` 参数会让 pandas 先将时间视为本地时区再转为 UTC，如果数据本身已经是 UTC，会导致时间偏移。

**解决方案**：不用 `utc=True`，而是先 `to_datetime()` 再 `tz_localize("UTC")`。

## 经验教训

1. **多数据源分析的第一步是统一时区**：在所有数据加载函数中强制转为 tz-aware UTC，不要依赖数据源的自带格式。
2. **tz_localize 不是 tz_convert**：tz-naive 数据要用 `tz_localize`（声明时区），已有 tz-aware 数据要用 `tz_convert`（转换时区），不要混用。
3. **`min()`/`max()` 会传播时区**：对 tz-aware 列做聚合会保持 tz-aware，确保所有输入一致。

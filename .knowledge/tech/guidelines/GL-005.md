## 背景

地面观测数据是气象分析的基础，但直接从各国气象服务获取需要分别对接不同协议（NOAA ISD、DWD CDC、CMA 数据中心等），门槛高、格式混乱。Meteostat 作为聚合层，统一封装了全球气象站的历史观测数据，通过 Python 库提供 Pandas 友好的接口，免 API key，适合科研和教育场景。本条目基于 2026-07-18 的实际测试验证。

## 内容

### 推荐做法（recommend）

#### 1. 数据源概览

| 维度 | 说明 |
|------|------|
| 数据来源 | NOAA、DWD、国家气象服务等多源聚合 |
| 访问方式 | Python 库 `meteostat`（基于 bulk data interface） |
| API key | 不需要 |
| 时间范围 | 历史数据（部分站点可追溯到 1900 年代） |
| 空间覆盖 | 全球气象站（密度不均，欧美密集） |
| 数据频率 | 小时、日、月、气候常态 |
| 授权 | 非商业免费，数据遵循各自原始协议 |

**安装**：
```bash
pip install meteostat
```

#### 2. 核心接口

```python
from meteostat import Point, Daily, Hourly, Stations
from datetime import datetime

# 三种数据获取方式
# 方式 1: 按经纬度（自动找最近站）
point = Point(39.9042, 116.4074, alt=70)

# 方式 2: 按站点 ID
# station_id = "54511"  # 北京站

# 方式 3: 查询附近站点
stations = Stations()
stations = stations.nearby(39.9042, 116.4074, radius=50000)
```

#### 3. 日数据获取模板

```python
from meteostat import Point, Daily
from datetime import datetime

# 北京
beijing = Point(39.9042, 116.4074, 70)
start = datetime(2026, 6, 1)
end = datetime(2026, 6, 30)

data = Daily(beijing, start, end)
df = data.fetch()
# df 包含: tavg, tmin, tmax, prcp, snow, wdir, wspd, wpgt, pres, tsun
```

#### 4. 小时数据获取模板

```python
from meteostat import Point, Hourly
from datetime import datetime

beijing = Point(39.9042, 116.4074, 70)
start = datetime(2026, 7, 1)

# df 包含: temp, dwpt, rhum, prcp, snow, wdir, wspd, wpgt, pres, tsun, coco
```

#### 5. 变量字段说明

| `prcp` | 日降水量 | mm |
| `snow` | 积雪深度 | mm |
| `wdir` | 平均风向 | 度 |
| `wspd` | 平均风速 | km/h |
| `wpgt` | 最大阵风 | km/h |
| `pres` | 海平面气压 | hPa |
| `tsun` | 日照时长 | 分钟 |

| 字段 | 含义 | 单位 |

| `pres` | 气压 | hPa |
| `coco` | 天气状况码 | 整数 |

#### 7. 数据完整性注意事项

- **缺失值**：用 `<NA>`（pandas NA）表示，不是 NaN，处理时需注意
- **站点密度**：中国地区站点稀疏，北京 50km 内仅 1 个站；欧美密集
- **数据延迟（2026-07-20 实测修正）**：**小时数据近乎实时**（延迟约 10-15 分钟），日数据延迟 0-2 天（详见第 13 节）。此前"1-7 天延迟"的说法仅适用于日数据，小时数据远比预期实时
- **历史长度**：不同站点起始时间不同，需实际查询确认
- **单位**：温度默认 °C，风速默认 km/h，需按需转换

#### 8. 大批量下载策略（2026-07-19 验证）

针对多站点、长时间范围的批量下载（如全国46个机场×2年），推荐以下策略：

**策略 1: 站点ID复用（关键优化）**

首次查询后保存 ICAO→station_id 映射，后续下载直接用 station_id 调用 `Daily(sid, ...)`，跳过 `Stations().nearby()` 查询，速度提升 3-5 倍。

```python
# 首次运行: 查询并保存映射
stations_metadata = []  # [{icao, station_id, station_name, ...}]

# 后续运行: 直接复用
for icao, sid in icao_to_sid.items():
    df = Daily(sid, start, end).fetch()
```

**策略 2: 逐级扩大半径搜索**

默认 30km 半径在中国西部地区常找不到站点。推荐逐级扩大：

```python
def find_station_with_expanding_radius(lat, lon):
    for radius_km in [30, 50, 100, 200, 500]:
        stations = Stations().nearby(lat, lon, radius=radius_km * 1000)
        df = stations.fetch()
        if len(df) > 0:
            return df.index[0], df.iloc[0]
    return None, None
```

**策略 3: 按ICAO号独立保存文件**

文件命名 `daily_{ICAO}.csv` / `hourly_{ICAO}.csv`，优势：
- 避免站点ID变更导致文件名混乱
- 按机场逻辑组织，便于业务使用
- 同一站点被多机场引用时（如北京首都/大兴共享54511），文件独立可追溯

**策略 4: 按年度分目录存储**

```
data/meteostat/
├── china_2025/     # 2025全年
│   ├── daily_ZBAA.csv
│   ├── hourly_ZBAA.csv
│   └── download_summary_2025.csv
└── china_2026/     # 2026年至今
    ├── daily_ZBAA.csv
    ├── hourly_ZBAA.csv
    └── download_summary.csv
```

#### 13. 数据实时性实测（2026-07-20）

对亚洲 8 站 + 美洲 10 站共 18 个站点进行实时性测试，测试时刻 2026-07-20 03:09 UTC（北京 11:09）。

**小时数据实时性**：**近乎实时，延迟仅约 12 分钟**

| 区域 | 测试站点数 | 最新小时数据 | 延迟 |
|------|-----------|-------------|------|
| 中国（北京/上海/杭州/广州） | 4 | 03:00 UTC | 0.2h（12min） |
| 东亚东南亚（东京/首尔/胡志明/曼谷） | 4 | 03:00 UTC | 0.2h（12min） |
| 美洲（美/加/阿根廷/智利/巴西/哥伦比亚/秘鲁） | 10 | 03:00 UTC | 0.2h（12min） |

所有 18 个站点的小时数据均更新到当前整点（03:00 UTC），延迟统一为 0.2 小时。Meteostat 通过 bulk data interface 近乎实时同步，**可作为准实时观测源使用**。

**日数据实时性**：延迟 0-2 天，存在地区差异

| 区域 | 最新日期 | 延迟(天) | 说明 |
|------|---------|---------|------|
| 美洲（美/加/阿根廷/智利/哥伦比亚/秘鲁） | 2026-07-20 | 0 | 当天即可获取 |
| 中国/越南/泰国 | 2026-07-19 | 1 | 次日更新 |
| 日本/韩国 | 2026-07-18 | 2 | 2 天延迟 |
| 巴西圣保罗（SBGR，异常） | 2026-07-13 | 7 | 日数据源更新异常 |

**关键发现**：
- 小时数据延迟远小于此前认知（原记"1-7 天延迟"实际仅适用于日数据）
- 与 NOAA ISD 原始数据（数月延迟）形成鲜明对比，Meteostat 聚合层接入了更实时的数据流
- 日数据延迟 0-2 天属正常（需等待全天观测结束并聚合）
- 个别站点（如巴西 SBGR）日数据延迟可达 7 天，疑数据源更新频率问题

### 禁止做法（avoid）

- ❌ 假设所有字段都有值（部分字段如 `wpgt`、`tsun` 常缺失）
- ❌ 用 `Point` 获取无人区数据（会返回最近站，可能距离很远）
- ❌ 不设置时间范围，全量拉取（数据量大，易超时）
- ❌ 混淆 `<NA>` 和 `NaN`，导致统计计算异常
- ❌ 用作风速、降水的业务决策（站点密度不足，精度有限）

## 适用场景

- 历史气候分析（单站或多站时间序列）
- 科研项目的地面观测数据获取
- 气象教育与可视化
- 站点数据初步探查（找站点、查可用性）
- 验证模式输出与观测的对比

## 不适用场景

- 严格实时业务预报（小时数据延迟约 12 分钟，非完全实时；日数据延迟 0-2 天）
- 高密度空间分析（站点稀疏，应使用再分析数据如 ERA5）
- 商业产品（需确认数据源授权）
- 中国区域精细分析（站点密度低，建议结合 CMA 数据）
